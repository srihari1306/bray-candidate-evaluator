1. Project Overview
1.1 What is being built
The Smart Interviewer is a new feature module added to the existing Resume Scanner platform. After a recruiter scans resumes and reviews the candidate dashboard, they can trigger a fully automated AI interview for any candidate directly from the dashboard. The system schedules the interview, emails the candidate, conducts a structured video interview through a Microsoft Teams side panel, records the session, transcribes and evaluates all answers using Azure AI, and writes scores back to the dashboard — all without recruiter involvement after the initial trigger.
1.2 What already exists (do NOT rebuild)
React JS frontend — resume scanner dashboard with tabular candidate analysis and scoring
FastAPI backend — resume parsing, Azure OpenAI integration for analysis, Azure AI Search for semantic matching
Azure OpenAI (GPT-4 + embeddings) — already provisioned and in use
Azure AI Search — already provisioned and in use
Azure Blob Storage — already provisioned and in use
Microsoft 365 / Teams tenant — organisation already has this
1.3 What is being added
New FastAPI routes for interview lifecycle management
Azure Communication Services (Email) — for candidate email delivery
Azure Speech Service — for real-time speech-to-text of candidate answers
Azure Cosmos DB — for interview sessions, transcripts, and scores
Teams Side Panel App (React + Teams JS SDK) — the interview UI inside the Teams meeting
In-browser recording via MediaRecorder API — captures mic + screen, uploads to Azure Blob
Dashboard updates — Start Interview button, status badges, results columns

2. Complete User & System Flow
2.1 Recruiter flow
Recruiter opens the Resume Scanner dashboard
Reviews the candidate table with resume analysis scores
Clicks "Start Interview" button in a candidate's row
A modal appears: recruiter selects date and time for the interview
Recruiter clicks Confirm — the system schedules the interview
Dashboard shows status badge: Scheduled for that candidate
When the interview completes, the dashboard auto-updates with interview scores, transcript link, and recording link
2.2 Candidate flow
Candidate receives an email with: their name, the interview time, a direct Teams meeting link (static link), and clear instructions to join at the scheduled time
At the scheduled time, candidate clicks the Teams link and joins the meeting
The Teams side panel opens automatically inside the meeting
Stage 1 — Setup: panel instructs candidate to turn on camera, unmute microphone, and share their entire screen. A Done button confirms completion
Recording begins automatically at this point (MediaRecorder captures mic + screen)
Stage 2 — Question 1: the first question appears as text on the panel. Candidate reads it. When ready, clicks the mic button. Speaks their answer. Clicks Done when finished. Answer is transcribed via Azure Speech
Question 2 appears. Same flow. Answer transcribed.
Question 3 appears. Same flow. Answer transcribed.
Stage 3 — Goodbye: panel shows a thank-you message. Recording stops. All data uploads to backend. Session closes.
2.3 System background operations
On interview schedule: FastAPI stores session in Cosmos DB, sends email via ACS
When candidate joins and panel loads: panel fetches session data and questions from FastAPI using session token
During interview: each answer audio is streamed to Azure Speech SDK → transcript returned to panel
On interview complete: panel POSTs all 3 transcripts to FastAPI, uploads recording blob to Azure Blob Storage via SAS URL
FastAPI calls Azure OpenAI to evaluate each answer → produces per-question scores and final composite score
Cosmos DB updated with all results → dashboard polling endpoint returns new data → UI updates automatically

6. FastAPI Backend — New Routes
6.1 Router file
Create a new file: backend/app/routers/interview.py. Register it in main.py with: app.include_router(interview.router, prefix="/interview", tags=["interview"])
6.2 Route specifications
POST /interview/schedule
Triggered by recruiter clicking Confirm in the Schedule Interview modal.
Request body: candidate_id, candidate_name, candidate_email, scheduled_time (ISO 8601), job_description_id
Actions: (1) Generate a UUID session ID. (2) Create Cosmos DB document with status=scheduled. (3) Send email to candidate via ACS with static Teams link, scheduled time, and session ID embedded in a URL param. (4) Return session ID and status to frontend.
Response: { session_id, status: "scheduled", teams_link }
GET /interview/session/{session_id}
Called by the Teams side panel on load to get session data and questions.
Actions: Fetch Cosmos DB document by session_id. Update status to in_progress. Return questions array and candidate name.
Response: { candidate_name, questions: [str, str, str], session_id }
GET /interview/speech-token
Called by Teams panel to get a short-lived Azure Speech token. This keeps the Speech key off the client.
Actions: Exchange AZURE_SPEECH_KEY for a 10-minute auth token via Azure Speech REST API. Return token and region.
Response: { token, region }
GET /interview/upload-url/{session_id}
Called by Teams panel just before uploading the recording blob.
Actions: Generate a time-limited SAS URL (write permission, 30-minute expiry) for the blob: interview-recordings/{session_id}.webm using the Azure Blob Storage SDK.
Response: { sas_url, blob_name }
POST /interview/answers
Called by Teams panel after all 3 questions are answered.
Request body: session_id, answers: [{ question_index, question_text, transcript }], recording_blob_name
Actions: (1) Save transcripts to Cosmos DB. (2) Trigger evaluation_service.evaluate_answers() asynchronously. (3) Return 202 Accepted immediately so panel does not wait.
POST /interview/evaluate (internal — called by answers handler)
This is not a public endpoint. It is an internal async function called after answers are received.
Actions: (1) For each answer, call Azure OpenAI GPT-4 with a scoring prompt. (2) Receive score (0-10) and reasoning per question. (3) Compute final_score as mean of 3 scores. (4) Update Cosmos DB document with all scores and status=completed.
GET /interview/results/{session_id}
Called by dashboard polling to check if results are ready.
Response: { status, final_score, answers: [{ score, score_reasoning, transcript }], recording_sas_url }
GET /interview/candidate/{candidate_id}
Called by dashboard on load to check if a candidate already has a completed interview session.
Response: Latest session document for this candidate, or null if none.

7.2 Email template sent to candidate
The email must include the following:
Subject: Interview Invitation — [Job Title] at [Company Name]
Candidate name (personalised greeting)
Scheduled date and time (in a human-readable format, include timezone)
Clear instructions: Join the Teams meeting at the scheduled time, ensure your camera and microphone work, be in a quiet environment
The static Teams meeting link as a prominent button/link
A note that the interview is conducted by an AI system
The session_id embedded as a URL parameter in the Teams link for panel identification, e.g.: https://teams.microsoft.com/l/meetup-join/...?context={session_id: "abc123"}
Important: The session_id must be appended to the Teams link so that when the candidate clicks the link and the panel loads, the panel can extract the session_id from the URL context and call GET /interview/session/{session_id} to load the right interview data.

8.4 App.jsx — stage router
The root component manages which stage is shown. Stages are: setup, question (with current question index), and goodbye.
import { useState, useEffect } from 'react';
import { app } from '@microsoft/teams-js';
import SetupStage from './stages/SetupStage';
import QuestionStage from './stages/QuestionStage';
import GoodbyeStage from './stages/GoodbyeStage';
import { useInterviewSession } from './hooks/useInterviewSession';
import { useMediaRecorder } from './hooks/useMediaRecorder';

export default function App() {
  const [stage, setStage] = useState('loading');
  const [questionIndex, setQuestionIndex] = useState(0);
  const { session, loadSession } = useInterviewSession();
  const { startRecording, stopRecordingAndUpload } = useMediaRecorder();

  useEffect(() => {
    app.initialize().then(() => {
      const context = app.getContext();
      const sessionId = extractSessionIdFromContext(context);
      loadSession(sessionId).then(() => setStage('setup'));
    });
  }, []);

  function handleSetupDone() {
    startRecording();   // begin MediaRecorder capture
    setStage('question');
  }

  function handleAnswerDone(transcript) {
    session.recordAnswer(questionIndex, transcript);
    if (questionIndex < 2) { setQuestionIndex(i => i + 1); }
    else { handleAllDone(); }
  }

  async function handleAllDone() {
    setStage('goodbye');
    await stopRecordingAndUpload(session.id);
    await session.submitAnswers();
  }

  if (stage === 'loading') return <div>Loading your interview...</div>;
  if (stage === 'setup') return <SetupStage onDone={handleSetupDone} candidateName={session.candidateName} />;
  if (stage === 'question') return <QuestionStage question={session.questions[questionIndex]} index={questionIndex} onDone={handleAnswerDone} />;
  if (stage === 'goodbye') return <GoodbyeStage />;
}

export function useMediaRecorder() {

9. Azure OpenAI — Answer Evaluation
9.1 Evaluation prompt design
For each answer, send the following prompt to GPT-4 via the existing Azure OpenAI endpoint:
system: |
  You are an expert interviewer and talent evaluator.
  Your task is to evaluate a candidate's spoken answer to an interview question.
  Evaluate on three criteria: Relevance (does it answer the question?),
  Clarity (is it well-structured and easy to understand?),
  Depth (does it demonstrate genuine knowledge or experience?).
  Return ONLY a JSON object with no other text:
  { "score": <integer 0-10>, "reasoning": "<one concise sentence>" }

user: |
  Interview Question: {question_text}
  Candidate Answer: {transcript}
9.2 Final score computation
After all three answers are evaluated, compute:
final_score = round((score_q1 + score_q2 + score_q3) / 3, 1)
9.3 Evaluation service (evaluation_service.py)
async def evaluate_answers(session_id: str, answers: list[AnswerInput]):
  scored = []
  for answer in answers:
    response = await openai_client.chat.completions.create(
      model=AZURE_OPENAI_DEPLOYMENT,
      messages=[{system_prompt}, {user_prompt(answer)}],
      temperature=0.2
    )
    result = json.loads(response.choices[0].message.content)
    scored.append({ ...answer, "score": result['score'], "score_reasoning": result['reasoning'] })

  final_score = round(sum(a['score'] for a in scored) / len(scored), 1)
  await cosmos_client.update_session(session_id, {
    'answers': scored,
    'final_score': final_score,
    'status': 'completed'
  })

10.4 useInterviewStatus.js hook
Polls GET /interview/results/{session_id} every 5 seconds for sessions in scheduled or in_progress state. Stops polling when status becomes completed or failed. Updates candidate row state in the parent component.
export function useInterviewStatus(sessionId, initialStatus) {
  const [status, setStatus] = useState(initialStatus);
  const [results, setResults] = useState(null);

  useEffect(() => {
    if (!sessionId || status === 'completed') return;
    const interval = setInterval(async () => {
      const data = await getResults(sessionId);
      setStatus(data.status);
      if (data.status === 'completed') {
        setResults(data);
        clearInterval(interval);
      }
    }, 5000);
    return () => clearInterval(interval);
  }, [sessionId, status]);

  return { status, results };
}

13. Important Notes for the Coding Assistant
13.1 Do not change existing code unless explicitly listed
The resume scanner's existing parsing logic, OpenAI analysis, Azure Search integration, and candidate scoring must not be modified. All new code is additive. The only existing files that need modification are CandidateTable.jsx (add Interview column) and main.py (register new router).
13.2 Session ID in Teams context
The session_id must travel from the backend email → the Teams meeting link as a URL parameter → the Teams panel context. In the Teams JS SDK, use app.getContext() to get the meeting context. The session_id should be appended to the Teams link as a custom parameter when the email is sent. In the panel, parse it from the subEntityId or meetingId context fields. Test this flow carefully as Teams context parameter passing can be tricky.
13.3 MediaRecorder browser support
MediaRecorder with video/webm is supported in Chromium-based browsers (Chrome, Edge). Teams desktop client uses a Chromium-based webview, so this works. Safari does not support video/webm — use a fallback to audio/webm if video fails. Test inside the actual Teams client, not just a regular browser, as Teams imposes additional permissions.
13.4 Speech SDK and microphone in Teams panel
The Azure Speech SDK's fromDefaultMicrophoneInput() will use the microphone that Teams is already using. The candidate must have already unmuted in Teams before this works. The SetupStage instructions to unmute first are therefore critical. If recognition returns empty results, the most common cause is the mic being muted at the Teams level.
13.5 CORS configuration
FastAPI must allow CORS from two origins: the React dashboard URL and the Teams panel Static Web App URL. During local development, also allow localhost:3000 and localhost:3001. Use the fastapi.middleware.cors CORSMiddleware.
13.6 Error handling
Every stage in the Teams panel must handle network failures gracefully. If GET /interview/session fails, show a friendly error and a retry button. If speech recognition fails, allow the candidate to re-record. If the upload fails, retry automatically up to 3 times before showing an error. The candidate must never be left on a blank screen.
13.7 Cosmos DB indexing
Add a composite index on (candidate_id, status) in Cosmos DB for efficient queries on GET /interview/candidate/{candidate_id}. Add a single-field index on status for admin queries. Configure this in the Cosmos DB portal under the container's Indexing Policy, or in cosmos_setup.py.
